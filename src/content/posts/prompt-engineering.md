# 提示词工程与模型原理笔记

## ⚙️ 模型核心采样参数

* **温度 (Temperature)** `0 ~ 2`
  * **低温（趋近于 0）**：输出更具确定性、更“冷漠”，适用于精确输出、代码生成等必须基于客观事实的场景。
  * **高温（趋近于 2）**：思维越发散、越有“人情味”，适用于对话交互、创意写作、头脑风暴等发挥想象力的场景。
* **Top-K**
  * **原理**：限制模型只从概率最高的前 `K` 个词中进行选择，超出范围的词直接舍弃。
  * **作用**：兜底过滤掉低概率的“垃圾词”，保证生成文本的基本合理性。
* **Top-P (核采样 / Nucleus Sampling)**
  * **原理**：设置一个累积概率阈值（例如 `0.9`）。模型将词汇按概率从大到小排序并累加，当累加值达到 90% 时停止。后续生成的词只在这些累积概率内的词中产生（如果前两个词的概率和已达 90%，则答案就在这两个词中产生）。

## 💡 提示词设计方法 (Prompt Engineering)

### 1. 样本驱动 (Few-Shot Tuning)

* **零样本提示 (Zero-Shot)**：不提供任何示例，直接向模型下达指令并要求回答。
* **单样本提示 (One-Shot)**：提供**一个**规范示例，引导模型模仿该示例的格式或语气进行回答。
* **少样本提示 (Few-Shot)**：提供**两个或以上**多样化的示例，更精准地规范模型的输出模式。

### 2. 思维链 (CoT - Chain of Thought)

* **原理**：引导模型一步步展示思考与推理过程，而非直接给出答案。
* **作用**：显著提升模型处理数学、逻辑、常识推理等复杂任务的能力。

## 🛠️ 技术原理与底层机制

### 文本分词 (Tokenization)

* **常用工具**：字节对编码（BPE）、SentencePiece 等。
* **核心作用**：将人类的自然语言转化为机器可识别的数字 ID (Tokens)。
* **实际影响**：分词效率直接影响 API 输入的计费成本以及上下文窗口（Context Window）的实际占用。

## ⚠️ 模型幻觉 (Hallucination)

### 1. 幻觉分类

* **事实性幻觉**：模型自信地给出了与真实世界物理规律或历史事实相违背的错误信息。
* **忠实性 / 上下文幻觉**：在提供了特定参考资料的情况下，模型的回答偏离或违背了给定的上下文。
* **逻辑 / 推理幻觉**：虽然输出的孤立事实是正确的，但事实之间的因果关系、推导逻辑是不正确的。

### 2. 应对幻觉的有效方法

* **调整采样参数**：适当降低 Temperature 或 Top-P 以提高确定性。
* **检索增强生成 (RAG)**：接入外部知识库，为模型提供实时、准确的背景参考。
* **提示词约束**：在 System Prompt 或 User Prompt 中显式限制其“不知道就说不知道，不要编造”。
* **工具调用 (Function Calling / Tool Use)**：允许模型调用计算器、搜索引擎、数据库查询等外部工具。
* **模型自我反思 (Self-Reflection)**：通过多轮对话或设定反思步骤，让模型在输出前自行检查和修正错误。
